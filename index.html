M5 Validator Governance and Protocol Upgrade Orchestration Architecture

Decentralized Governance and Protocol Evolution for Live Institutional Validator Infrastructure
Phase: M5: Live Validator Operations Classification: Governance Architecture Specification Version: 1.0, March 2026 Alignment: Superstructure Validator Platform / Post Fiat L1 Mainnet
Document Purpose and Scope This specification defines the complete governance and protocol upgrade orchestration architecture for the Post Fiat Superstructure validator platform. It extends the governance signal propagation model in M5 Cross-Operator Coordination (Section 5) with full lifecycle detail: proposal origination through ratification, heterogeneous operator upgrade coordination, rollback architecture, middleware vote propagation mechanics, and emergency upgrade paths. All governance mechanisms preserve the zero-custody boundary established in M2 and the Maker/Checker/Reviewer institutional control model established in M3. Middleware aggregates and routes governance signals but holds no voting authority and cannot determine outcomes unilaterally.

Governance Proposal Lifecycle

A governance proposal is a formal request to change network parameters, operator membership, protocol behavior, or governance rules. Every change to the validator network that affects more than one operator or any institutional custody arrangement must go through the proposal lifecycle defined here. Informal agreements between operators do not constitute governance decisions and carry no binding weight.

1.1 Origination Rules

Proposals originate from two sources: OCP (for protocol-level changes including upgrades, parameter adjustments, and security patches) and institutions (for operator-level changes including additions, removals, and custody policy updates). Operators have no direct proposal origination rights. An operator who believes a change is warranted submits a formal request to OCP through the admin channel; OCP evaluates whether the request meets the criteria to become a formal proposal. This prevents operators from constructing self-serving proposals and routing them directly to institutions.

1.2 Proposal Format Schema

Every proposal submitted to the governance pipeline must conform to the following schema. OCP validates schema compliance before broadcasting. A non-conforming proposal is returned to the originator with a validation error and does not enter the governance pipeline.

Governance Proposal Payload Schema (JSON, RFC 8785 canonical)

The rollback_ref field is mandatory for all PROTOCOL_UPGRADE proposals. An upgrade proposal without a designated rollback binary hash does not pass OCP validation. This ensures institutions authorize both the upgrade and its rollback path in a single ceremony, which is what prevents synchronized signing requirements during a rollback event.

1.3 Discussion Period Mechanics

The discussion period is the window between proposal broadcast and voting window open. Its purpose is to give institutions time to evaluate the proposal internally, request clarifications from OCP, and prepare their M/C/R process before the voting window begins. Operators cannot participate in or observe the discussion period; it is an institution-only process mediated through the institutional gateway.

Proposal Immutability After Broadcast A proposal cannot be modified after OCP broadcasts it. Withdrawal resets the full lifecycle including discussion and voting periods. Amendment by substitution is the only permitted amendment mechanism. In-place editing of a live proposal is not permitted because institutions may have already begun their internal M/C/R evaluation against the original content hash.

Protocol Upgrade Coordination Across Heterogeneous Operator Infrastructure

Operator infrastructure is heterogeneous by design: operators choose their own hardware, cloud regions, operating systems, and deployment tooling. Protocol upgrades must propagate across this diverse fleet without requiring operators to standardize their infrastructure or synchronize their maintenance windows with each other. OCP is the only coordination point; operators do not coordinate upgrade timing directly.

2.1 Version Discovery

OCP maintains the authoritative version registry: a record of every binary version currently running across all operator nodes, derived from the secure boot attestation data emitted through the telemetry channel. Operators do not report versions directly; OCP infers running version from binary hash comparison against the known manifest registry. This means version discovery is passive for operators: they do not need to implement a version reporting API, and OCP does not need to query operators for their current version.

2.2 Compatibility Negotiation

Compatibility is OCP-determined, not operator-negotiated. When a new version is ratified through the governance process, OCP publishes a compatibility matrix in the broadcast payload. The matrix specifies which existing versions can process signing ceremonies from nodes on the new version and vice versa. This allows mixed-version fleets to remain operational during staged rollouts.

2.3 Staged Rollout Sequencing and Operator-Specific Upgrade Windows

Staged rollout uses a three-wave model as specified in M5 Cross-Operator Coordination Section 3.1, with the following additions governing heterogeneous operator infrastructure specifically.

Wave assignment is not strictly deterministic: OCP uses a scoring function that incorporates operator health score, patch currency signal, infrastructure diversity (to avoid correlated failure across operators on similar hardware configurations), and time zone distribution (to avoid concentrating early waves in a single region, which would leave other regions without recently-upgraded nodes during the observation window). An operator can request a wave deferral through the admin channel with a documented maintenance conflict. OCP may grant up to a 48-hour deferral for Wave 2 or Wave 3; Wave 1 deferrals require OCP engineering approval because Wave 1 failures trigger rollback decisions.

Operator-specific upgrade windows are declared by the operator through the admin channel: a 4-hour window within which the operator is prepared to receive and apply the binary. OCP schedules the binary push within that window. If an operator fails to declare a window within 24 hours of receiving a wave assignment notification, OCP assigns a default window during the operator's lowest-ceremony-volume period as inferred from their telemetry history. Operators running across multiple time zones may declare multiple acceptable windows; OCP selects the earliest window that falls within the observation schedule.

No Synchronized Signing Window Requirement The staged rollout design ensures that at least 50% of the fleet remains on the previous authorized version throughout any upgrade wave. Signing ceremonies route to operators on either version during the transition period (subject to compatibility matrix constraints). No institution is asked to schedule, hold, or defer a signing ceremony because of an upgrade in progress on another operator's node. The institution signed for both the upgrade binary and the rollback binary in Wave 0 before any operator received the new binary. The signing window requirement is front-loaded into governance ratification, not distributed across operational upgrade events.

Rollback Architecture

Rollback architecture addresses the failure case where an upgrade introduces a defect that is not caught by pre-upgrade testing and manifests on live infrastructure. The goal is to restore the fleet to a known-good state as quickly as possible while preserving consensus continuity and ensuring no validator produces conflicting outputs during the transition.

3.2 State Checkpoint Preservation

Before any upgrade binary is applied to a node, OCP instructs the node to create a state checkpoint. The checkpoint captures the configuration state, nonce log depth, ceremony queue state, and binary hash at the moment immediately before upgrade application. This checkpoint is uploaded to the OCP forensic store over a one-way transfer (the store cannot write back to the node) and registered with a content hash in the OCP audit log.

State checkpoint creation is a prerequisite gate for upgrade application: OCP does not push the new binary to a node until it receives confirmation that the checkpoint upload completed successfully. An operator cannot skip the checkpoint step. If checkpoint creation fails (for example, due to insufficient disk space or a telemetry outage), the node is excluded from the current wave and placed in the next wave after the operator resolves the underlying issue.

On rollback, the state checkpoint is used to verify post-rollback state rather than to restore state directly. The rollback binary is applied fresh; after application, OCP verifies that the node's configuration hash and nonce log depth match the pre-upgrade checkpoint values. A discrepancy between post-rollback state and checkpoint state generates a P2 alert and blocks the node from re-entering ceremony operations until the discrepancy is resolved. This matters because a node that processed ceremonies on the new binary before rollback may have advanced its nonce log; OCP accounts for this by accepting nonce log depths at or above the checkpoint value, not requiring an exact match.

3.3 Partial-Fleet Rollback Isolation

A partial-fleet rollback isolates operators in the current wave from operators in completed waves or earlier waves that are not yet affected. This prevents an upgrade defect that manifests only on certain hardware configurations from triggering a rollback of operators on configurations where the binary is running correctly.

Partial-fleet rollback applies only when the trigger is wave-specific and does not involve consensus divergence or a P1 incident. Consensus divergence and P1 incidents always trigger fleet-wide rollback because the risks of a split fleet outweigh the benefit of preserving some operators on the new version. For wave-specific health degradation triggers, OCP may isolate the rollback to the affected wave if: the non-affected waves have been stable for at least 4 hours, the compatibility matrix permits mixed-version operation, and no ceremony routing disruption results from leaving some operators on the new version while rolling others back.

An operator whose wave is rolled back is not automatically moved to a later wave for the next upgrade attempt. OCP conducts a post-rollback analysis to identify whether the failure was hardware-specific, configuration-specific, or binary-level. Hardware and configuration failures result in the operator receiving targeted remediation guidance before being re-queued. Binary-level failures result in fleet-wide rollback regardless of isolation status.

3.4 Rollback Verification Procedures and Consensus Continuity

Rollback verification confirms that every node in the rollback scope has successfully returned to the authorized rollback binary and is processing ceremonies correctly. Verification runs on a 5-minute post-rollback observation window per node.

Consensus continuity during rollback is maintained by suspending new signing ceremony initiation for nodes in transition. Nodes on stable versions (not in the rollback scope, or already confirmed rolled back) continue processing ceremonies. OCP routes ceremonies away from transitioning nodes using the health score tier mechanism: a node applying the rollback binary enters the equivalent of a planned maintenance state, and its routing weight drops to 0 until verification completes. This is the same mechanism used for planned maintenance handoffs and does not require any institution ceremony action.

Governance Vote Propagation Through Middleware

Middleware (OCP) handles vote collection, aggregation, and result certification. This concentrated role creates an obvious risk: a middleware that can manipulate votes can determine governance outcomes. The architecture addresses this risk by making OCP's role in vote processing verifiable and auditable by every institution independently. OCP cannot manufacture a valid signed vote envelope because it does not hold institutional signing keys. OCP can only count what institutions actually sign.

4.1 Vote Collection

Institutions submit votes as signed vote envelopes through the institutional gateway. Each vote envelope is cryptographically bound to a specific proposal and cannot be reused across proposals.

Vote Envelope Schema
vote_id: UUID v4 // OCP-assigned on receipt; returned to institution as receipt
proposal_id: UUID v4 // Must match a live proposal in VOTING state
institution_id: string // Validated against OCP institution registry
vote_direction: enum // YES | NO | ABSTAIN
proposal_hash: SHA-256 // Institution's independent hash of the proposal payload; must match OCP-broadcast hash
voted_at: ISO8601 UTC // OCP NTP-sourced; institution timestamp also included for audit comparison
nonce: UUID v4 // Single-use nonce; checked against 30-day retention log
institution_sig: Ed25519 // Signed inside institution custody environment; Reviewer-authorized per M/C/R

The proposal_hash field requires the institution to independently compute the SHA-256 hash of the proposal payload they received and include it in the vote envelope. OCP verifies this hash matches the hash of the proposal it broadcast. A mismatch means the institution received a tampered proposal and voted on different content than the rest of the cohort. OCP rejects the vote, notifies the institution, and generates a P1 security alert for investigation of the tamper event. This mechanism ensures that vote fraud by proposal substitution is detectable.

4.2 Vote Aggregation

OCP aggregates votes as they are received. The running tally is not disclosed to any party before the voting deadline. After the deadline, OCP publishes the final tally. The aggregation process itself is verifiable: every institution can confirm that its own vote envelope was counted by requesting a vote receipt from OCP after the deadline. The receipt confirms that the institution's vote was received and included in the tally without revealing how any other institution voted.

4.3 Quorum Calculation

Quorum calculation uses the active institution denominator defined in Section 1.4: institutions with at least one validator in healthy or degraded tier at voting window open. OCP records the denominator value at the moment the voting window opens and uses that value throughout the vote. The denominator does not change if an institution's validator enters a worse tier during the voting period; the institution retains its vote eligibility for the duration of the window.

Two quorum conditions must both be satisfied for a proposal to pass. First, the participation condition: the fraction of active institutions that submitted any valid vote (YES, NO, or ABSTAIN) must meet or exceed 50%. This prevents a proposal from passing with a technically valid supermajority that represents only a small subset of the cohort because most institutions did not participate. Second, the approval condition: the fraction of yes votes among all participating institutions must meet the proposal's quorum type threshold.

If the participation condition is not met, the proposal lapses regardless of how the votes that were cast broke down. A proposal where 60% of institutions vote YES but only 40% participated does not pass: the cohort is treated as having insufficiently engaged, not as having approved.

4.4 Result Certification

OCP generates a signed result certificate after the voting deadline. The certificate is delivered to all institutions via the institutional gateway and to all operators via the broadcast channel. It constitutes the authoritative record of the governance outcome and is the document that institutions reference when authorizing implementation ceremonies.

Result Certificate Schema

The result certificate includes the list of participating institution identifiers but not their vote directions. Any institution can verify that its own identifier appears in participating_institutions to confirm its vote was counted. No institution can determine how another institution voted from the certificate alone. The OCP audit store retains the full vote envelope set (including vote directions) under access control: the individual vote direction data is accessible only to OCP engineers under the dual-approval workflow and to institutions requesting their own vote record.

4.5 Audit Trail Generation and Middleware Authority Constraints

Every step of the vote lifecycle generates an immutable audit log entry: proposal broadcast, clarification requests, amendment withdrawals, vote receipts, deadline closure, tally computation, and result certificate issuance. Log entries are written to the OCP immutable audit store with SHA-256 hash chaining per M5 Security Hardening Section 4.1. The complete audit trail for any proposal is available to all institutions on request through the institutional gateway.

What Middleware Cannot Do OCP cannot manufacture a valid vote envelope: institutional signing keys never enter the Middleware Layer. OCP cannot change a vote direction after receipt: vote envelopes are write-once in the audit store. OCP cannot exclude a valid vote from the tally without the exclusion appearing in the audit log with a reason code visible to the affected institution. OCP cannot modify the quorum denominator after the voting window opens: the denominator is locked at window-open and recorded in the audit store. OCP cannot declare a result that does not follow from the signed vote envelopes: result certification is deterministic given the vote envelope set and the proposal's quorum type. Any institution can independently recompute the result from the audit trail and compare it against the OCP-certified result. A discrepancy would constitute evidence of middleware manipulation and is the basis for a governance dispute.

Emergency Upgrade Paths Preserving Maker/Checker/Reviewer Separation

Emergency upgrades address situations where a security vulnerability is being actively exploited or where a network stability threat requires faster action than the standard governance timeline permits. The design challenge is real: time pressure is the natural enemy of the M/C/R process. The solution is not to compress M/C/R but to compress everything else: the discussion period, the voting window, and the quorum threshold. The internal institutional process that leads to a signing decision stays intact.

5.1 Emergency Fast-Track Criteria

The emergency fast-track is not available on demand. OCP must classify a proposal as emergency and document the classification publicly before any compressed timeline applies. Classification criteria are narrow to prevent misuse.

CriterionRequired EvidenceDocumentation RequirementChallengeable By

Emergency Classification Is Not Self-Certifying OCP cannot invoke the emergency fast-track by asserting that a threat exists. The threat justification document must contain verifiable evidence: either a P1 incident record number from the M5 Security Hardening audit log, or network telemetry data from the OCP metric store with timestamps. Institutions receive the evidence simultaneously with the emergency proposal. An institution that believes the emergency classification is unjustified may submit a formal objection through the gateway. The objection does not block the emergency vote, but if the emergency vote passes and the post-emergency audit (Section 5.3) finds the classification was unjustified, the result is treated as a governance violation with consequences for OCP's operational authorization.

The reduced quorum threshold is a compensating recognition that some institutions will not be able to convene their Reviewer in 4 hours, not a signal that their participation is unimportant. Institutions that cannot participate in the emergency vote are notified of the outcome through the institutional gateway and have 72 hours to submit a dissenting record to the governance audit log. A dissenting record does not reverse the outcome but is part of the permanent governance history.

5.3 Maker/Checker/Reviewer Separation Under Emergency Conditions

The M/C/R process inside the institution's custody environment does not change for emergency upgrades. What changes is the external timeline that the institution is working against. The internal process still requires: a Maker to prepare the vote envelope and (if the proposal passes) the implementation ceremony payload, a Checker to verify the payload against the proposal content hash, and a Reviewer to authorize the signing action. Collapsing any two of these roles is not permitted under emergency conditions.

Two institution-side mechanisms support emergency M/C/R execution within compressed timelines.

First, backup Reviewer designation is required during institution enrollment. Every institution must name at least one backup Reviewer whose contact information is held by OCP for emergency notification. When OCP broadcasts an emergency proposal, it simultaneously sends direct emergency notifications to both the primary and backup Reviewer for each institution through the institutional gateway. The primary Reviewer is not given a head start; both are notified at the same time to maximize the probability that at least one is reachable within the 4-hour window.

Second, pre-authorized ceremony envelope templates are available for the most common emergency upgrade types (binary security patch, critical parameter change). During institution enrollment, the institution's Reviewer signs a meta-authorization that designates specific proposal types as eligible for expedited M/C/R: the Maker prepares the envelope, the Checker verifies it, and the Reviewer verifies only that the proposal type matches the pre-authorized category and that the content hash is valid. This reduces the Reviewer's review burden for the specific scenario the meta-authorization covers without waiving the Reviewer's participation. The meta-authorization itself requires a full M/C/R ceremony during enrollment and can be revoked by a subsequent Reviewer authorization.

5.4 Post-Emergency Audit Requirements

Every emergency governance action triggers a mandatory post-emergency audit within 72 hours of the emergency result being certified. The audit is not optional and cannot be waived by any party.

5.5 Abuse Prevention Mechanisms

Four structural mechanisms prevent the emergency fast-track from being used as a routine governance shortcut.

Frequency cap: No more than two emergency fast-track proposals may be active or completed within any rolling 90-day window. A third emergency proposal within 90 days requires unanimous institution approval to receive emergency classification. OCP cannot override this cap.

Mandatory evidence publication: Emergency classification requires simultaneous publication of the threat justification document to all institutions. A classification that cannot be supported with publishable evidence cannot proceed under emergency rules. This prevents OCP from invoking emergency classification based on information it is unwilling to share with the cohort.

Retrospective reversal right: Any emergency action that fails retrospective ratification (Section 5.4) is reversed by OCP within 48 hours of the ratification vote result. Reversal means applying the rollback binary (which was authorized as part of the original emergency proposal) and returning network parameters to pre-emergency values. The reversal itself is not a governance action; it is an automatic consequence of ratification failure, and OCP executes it without a new vote.

Governance violation consequence: A finding that an emergency classification was unjustified (from the OCP internal audit in Section 5.4) is recorded as a governance violation in the OCP operational record. Three governance violations within 24 months trigger a mandatory governance proposal to review OCP's role in the Superstructure architecture. That review proposal follows the standard governance lifecycle and cannot itself be classified as emergency.
